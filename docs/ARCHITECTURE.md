# Architecture

A high-altitude tour of how DockControl is wired. Read this before changing how services talk to each other.

## Services

A standard install is five containers in one Compose project (`docker-compose.yml`):

| Service | Image | Port (host) | Purpose |
| --- | --- | --- | --- |
| `postgres` | `postgres:16.6-alpine` | `127.0.0.1:5432` | Primary store. Holds users, projects, applications, deployments, system settings, agent tasks, audit log, notifications. |
| `caddy` | `caddy:2.9-alpine` | `80`, `443`, `443/udp` | Reverse proxy, auto-Let's Encrypt. Reads `.dockcontrol/reverse-proxy/Caddyfile`, regenerated by the API. |
| `api` | built from `apps/api/Dockerfile` | `4000` (bind via `API_BIND`) | NestJS backend. Talks to the host Docker daemon via `/var/run/docker.sock` to deploy user apps. |
| `dashboard` | built from `apps/dashboard/Dockerfile` | `3000` (bind via `DASHBOARD_BIND`) | Next.js 15 frontend. `NEXT_PUBLIC_API_URL` is inlined at build time. |
| `sftp` | built from `infra/sftp` | `2222` | Hardened OpenSSH for per-app SFTP access (chrooted users, bind-mounted app data). |

`postgres` binds to `127.0.0.1` only — it is never reachable from the public internet. `caddy`, `api` and `dashboard` are on a shared bridge network `dockcontrol-apps` so Caddy can proxy user-deployed app containers by `container_name` without publishing host ports. (A `redis` container used to ship with the stack but was removed — the API never consumed it.)

## LOCAL vs MULTI mode

`SystemSetting.deployment_mode` toggles the routing model.

- **LOCAL** (default). The API treats `project.serverId` as `127.0.0.1` and runs `docker compose` directly on the API host. Single VPS, simplest case.
- **MULTI**. Each project is bound to a `Server` row. For the local server (`host ∈ {127.0.0.1, localhost, ::1}`) the API runs Compose locally; otherwise it enqueues a typed `AgentTask` row and lets the remote `dockcontrol-agent` (Go binary, installed via `/api/agent/install.sh?token=…`) pick it up via long-poll.

Tasks are claimed with `SELECT … FOR UPDATE SKIP LOCKED`, which means two agents sharing the same token can never both grab the same task — required the day you add HA, present from day one. Synchronous flows (`LOGS`, `EXEC`) use `enqueueAndWait` with a timeout; async ones (`DEPLOY`, `START`, `STOP`, `RESTART`, `REMOVE`, `FILE_READ`, `FILE_WRITE`, `STATUS`) fire-and-forget and the agent reports back.

## On-disk layout: `.dockcontrol/`

The API persists everything operational under `.dockcontrol/` at the install root (default `/opt/dockcontrol/.dockcontrol/`). The same directory is bind-mounted into the `api` container at `/app/.dockcontrol` AND into the `caddy` container's `Caddyfile` path, so file edits from the API are visible to Caddy and vice versa.

```
.dockcontrol/
  apps/
    <slug-shortid>/              # one folder per deployed app
      docker-compose.yml         # generated by the API
      .dockcontrol.env             # managed env vars (denylisted from the file manager)
      <user source files…>
  databases/
    <slug>/                      # database stacks (Postgres, MySQL, MongoDB, …)
      docker-compose.yml
      data/                      # bind-mounted by the DB container
  mail/
    <domain>/                    # mail server (docker-mailserver: Postfix+Dovecot+rspamd) configs + data
      docker-compose.yml
  reverse-proxy/
    Caddyfile                    # regenerated on every domain change
  update.log                     # log of the most recent update.sh run (reset each run;
                                 # the API tails it for the dashboard updates page)
  .update-running                # marker-file mutex the API holds while update.sh runs
```

Every project also gets a dedicated user-defined bridge network `dockcontrol_proj_<slug>`. All apps in the same project join it via `networks: { dockcontrol_project: { external: true, name: dockcontrol_proj_<slug> } }`, so service-to-service DNS just works (`http://api/`, `http://worker/`, …) without exposing ports.

## Security model

- **JWT + refresh rotation.** Login mints a short-lived access token and a long-lived refresh token. Each refresh rotates both — a stolen refresh token is invalidated the next time the real user refreshes. Refresh tokens are persisted hashed (bcrypt) and revocable.
- **AES-256-GCM at rest.** `ENCRYPTION_KEY` (≥32 hex chars, i.e. 16 random bytes; the actual 32-byte AES key is derived via scrypt) encrypts secrets stored in `SystemSetting` (SMTP password, GitHub webhook secret, etc.) and credential fields on `Server`, `GitProvider`, `Database` rows. The envelope is `{ __sec: 1, v: '<aes-blob>' }` so a DB dump never reveals plaintext.
- **Encrypted backups.** Independent `BACKUP_ENCRYPTION_KEY` (separate from `ENCRYPTION_KEY` so backup access can be siloed from app secrets). When set, backup dumps are encrypted AES-256-GCM on the way to disk; SHA-256 is recorded and verified on restore.
- **RBAC.** Two role axes, both enforced server-side:
  - Global `Role`: `SUPERADMIN`, `ADMIN`, `USER`, `VIEWER`. First user becomes `SUPERADMIN` automatically.
  - Per-project `ProjectRole`: `OWNER`, `ADMIN`, `DEVELOPER`, `VIEWER`. The project membership grant is what gates access to deploys, env vars, logs, exec.
- **2FA (TOTP).** Per-user opt-in (or enforced by the admin policy). Standard RFC 6238 TOTP; recovery codes generated at enrolment, single-use, bcrypt-hashed at rest.
- **Webhook HMAC on raw body.** GitHub / GitLab / Bitbucket payloads are verified against the *raw* request bytes (not the JSON-reparsed form) using a secret stored encrypted in `SystemSetting`. Bitbucket payloads without a signature header are rejected — `x-event-key` alone is not authentication.
- **Shell-free exec.** All `exec` paths use `execFile` with argv arrays. No shell interpolation, ever. File-task payloads reject any path component containing `..`.
- **Git tokens.** Injected for the duration of one clone via `http.extraheader`, never written to `.git/config`. Cleared from the process env at the end of the spawn.

## Notification flow

`NotificationsService` is the single fan-out. Events that produce notifications: deployment outcomes (`deployOk` / `deployFail`), server offline (`serverOff`), SSL certificate expiry/renewal failures (`sslExpire`), backup outcomes (`backupOk` / `backupFail`), and server alerts (CPU/RAM/disk threshold rules). Email delivery of each event is filtered per-recipient against the user's notification preferences (`User.notificationPrefs`, edited under Settings → Notifications — opt-out model, absent = allowed); the in-app feed is written regardless.

Each event renders **once** into a structured payload, then dispatches to the operator's configured channels:

- **SMTP** — read from `SystemSetting.smtp_*` (see [CONFIG.md](./CONFIG.md)). Reused transport, hot-reloaded when settings change via `SystemConfigService.onChange()`.
- **Webhook** — arbitrary URL the operator configures (e.g. Discord, Slack, n8n, custom). POSTed as plain JSON.
- **In-app feed** — every global event also persists one `Notification` row per active `ADMIN`/`SUPERADMIN` (the admin list is cached 60 s; deploy outcomes additionally target the triggering user). The dashboard header bell polls `GET /notifications/unread-count` every 30 s and reads the feed via `GET /notifications` (`POST /notifications/:id/read`, `POST /notifications/read-all` for read state). Rows older than 90 days are pruned best-effort at creation time. Works even when SMTP/webhooks are unconfigured.

If no outbound channel is configured, the event is still logged on the API side and lands in the in-app feed. Planned (not yet implemented): per-target webhook HMAC signatures, security events (new login from unknown IP, 2FA disabled).
