# Architecture

A high-altitude tour of how Kryptalis is wired. Read this before changing how services talk to each other.

## Services

A standard install is five containers in one Compose project (`docker-compose.yml`):

| Service | Image | Port (host) | Purpose |
| --- | --- | --- | --- |
| `postgres` | `postgres:16-alpine` | `127.0.0.1:5432` | Primary store. Holds users, projects, applications, deployments, system settings, agent tasks, audit log. |
| `redis` | `redis:7-alpine` | `127.0.0.1:6379` | Cache + job queue. Loopback only. |
| `caddy` | `caddy:2-alpine` | `80`, `443`, `443/udp` | Reverse proxy, auto-Let's Encrypt. Reads `.kryptalis/reverse-proxy/Caddyfile`, regenerated by the API. |
| `api` | built from `apps/api/Dockerfile` | `4000` | NestJS backend. Talks to the host Docker daemon via `/var/run/docker.sock` to deploy user apps. |
| `dashboard` | built from `apps/dashboard/Dockerfile` | `3000` | Next.js 15 frontend. `NEXT_PUBLIC_API_URL` is inlined at build time. |

`postgres` and `redis` bind to `127.0.0.1` only — they are never reachable from the public internet. The other three are on a shared bridge network `kryptalis-apps` so Caddy can proxy user-deployed app containers by `container_name` without publishing host ports.

## LOCAL vs MULTI mode

`SystemSetting.deployment_mode` toggles the routing model.

- **LOCAL** (default). The API treats `project.serverId` as `127.0.0.1` and runs `docker compose` directly on the API host. Single VPS, simplest case.
- **MULTI**. Each project is bound to a `Server` row. For the local server (`host ∈ {127.0.0.1, localhost, ::1}`) the API runs Compose locally; otherwise it enqueues a typed `AgentTask` row and lets the remote `kryptalis-agent` (Go binary, installed via `/api/agent/install.sh?token=…`) pick it up via long-poll.

Tasks are claimed with `SELECT … FOR UPDATE SKIP LOCKED`, which means two agents sharing the same token can never both grab the same task — required the day you add HA, present from day one. Synchronous flows (`LOGS`, `EXEC`) use `enqueueAndWait` with a timeout; async ones (`DEPLOY`, `START`, `STOP`, `RESTART`, `REMOVE`, `FILE_READ`, `FILE_WRITE`, `STATUS`) fire-and-forget and the agent reports back.

## On-disk layout: `.kryptalis/`

The API persists everything operational under `.kryptalis/` at the install root (default `/opt/kryptalis/.kryptalis/`). The same directory is bind-mounted into the `api` container at `/app/.kryptalis` AND into the `caddy` container's `Caddyfile` path, so file edits from the API are visible to Caddy and vice versa.

```
.kryptalis/
  apps/
    <slug-shortid>/              # one folder per deployed app
      docker-compose.yml         # generated by the API
      .kryptalis.env             # managed env vars (denylisted from the file manager)
      <user source files…>
  databases/
    <slug>/                      # database stacks (Postgres, MySQL, MongoDB, …)
      docker-compose.yml
      data/                      # bind-mounted by the DB container
  mail/
    <domain>/                    # mail server (Mailcow / Stalwart) configs + data
      docker-compose.yml
  reverse-proxy/
    Caddyfile                    # regenerated on every domain change
  update.log                     # last 500 lines of update.sh
  update-status.json             # current update state (read by the dashboard)
  auto-update.pref               # "disabled" string when operator turned auto-update off
  api-etag, api-cached-sha       # GitHub API caching state for update.sh
```

Every project also gets a dedicated user-defined bridge network `kryptalis_proj_<slug>`. All apps in the same project join it via `networks: { kryptalis_project: { external: true, name: kryptalis_proj_<slug> } }`, so service-to-service DNS just works (`http://api/`, `http://worker/`, …) without exposing ports.

## Security model

- **JWT + refresh rotation.** Login mints a short-lived access token and a long-lived refresh token. Each refresh rotates both — a stolen refresh token is invalidated the next time the real user refreshes. Refresh tokens are persisted hashed (bcrypt) and revocable.
- **AES-256-GCM at rest.** `ENCRYPTION_KEY` (32 bytes / 16 hex pairs) encrypts secrets stored in `SystemSetting` (SMTP password, GitHub webhook secret, etc.) and credential fields on `Server`, `GitProvider`, `Database` rows. The envelope is `{ __sec: 1, v: '<aes-blob>' }` so a DB dump never reveals plaintext.
- **Encrypted backups.** Independent `BACKUP_ENCRYPTION_KEY` (separate from `ENCRYPTION_KEY` so backup access can be siloed from app secrets). When set, backup dumps are encrypted AES-256-GCM on the way to disk; SHA-256 is recorded and verified on restore.
- **RBAC.** Two role axes, both enforced server-side:
  - Global `Role`: `SUPERADMIN`, `ADMIN`, `USER`, `VIEWER`. First user becomes `SUPERADMIN` automatically.
  - Per-project `ProjectRole`: `OWNER`, `ADMIN`, `DEVELOPER`, `VIEWER`. The project membership grant is what gates access to deploys, env vars, logs, exec.
- **2FA (TOTP).** Per-user opt-in (or enforced by the admin policy). Standard RFC 6238 TOTP; recovery codes generated at enrolment, single-use, bcrypt-hashed at rest.
- **Webhook HMAC on raw body.** GitHub / GitLab / Bitbucket payloads are verified against the *raw* request bytes (not the JSON-reparsed form) using a secret stored encrypted in `SystemSetting`. Bitbucket payloads without a signature header are rejected — `x-event-key` alone is not authentication.
- **Shell-free exec.** All `exec` paths use `execFile` with argv arrays. No shell interpolation, ever. File-task payloads reject any path component containing `..`.
- **Git tokens.** Injected for the duration of one clone via `http.extraheader`, never written to `.git/config`. Cleared from the process env at the end of the spawn.

## Notification flow

`NotificationsService` is the single fan-out. Events that produce notifications: deployment outcomes (success/failure), SSL renewal failures, server alerts (CPU/RAM/disk threshold), backup failures, security events (new login from unknown IP, 2FA disabled, registration approval needed).

Each event renders **once** into a structured payload, then dispatches to the operator's configured channels:

- **SMTP** — read from `SystemSetting.smtp_*` (see [CONFIG.md](./CONFIG.md)). Reused transport, hot-reloaded when settings change via `SystemConfigService.onChange()`.
- **Webhook** — arbitrary URL the operator configures per-project (e.g. Discord, Slack, n8n, custom). POSTed as JSON with HMAC if a per-target secret is set.

If neither is configured, the event is logged and shown in the dashboard's in-app notification feed; nothing is dropped silently.
